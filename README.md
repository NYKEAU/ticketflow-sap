# TicketFlow SAP

> Internal ticket management application built with **ABAP RAP (managed)**, **CDS Views**, **OData V4**, and **SAP Fiori Elements**.  
> Portfolio project — demonstrates modern ABAP development practices on SAP BTP ABAP Environment.

---

## Overview

TicketFlow SAP is a full-stack SAP application that allows teams to create, assign, track, and close internal support tickets through a clean Fiori Elements interface.

The project covers the full RAP stack from the database table to the published OData V4 service, including business logic, validations, and custom actions.

---

## Tech Stack

| Layer | Technology |
|---|---|
| Database | DDIC Transparent Table (`ZTICKETSAP`) |
| Data Model | CDS View Interface + Projection (`ZTICKET_I`, `ZTICKET_C`) |
| Business Logic | RAP Behavior Definition + Handler Class (`ZBP_TICKET_I`) |
| API | OData V4 (SAP RAP managed service) |
| UI | SAP Fiori Elements — List Report + Object Page |
| Platform | SAP BTP ABAP Environment |

---

## Features

- **Create / Edit / Delete** tickets with title, description, priority, category, and assignment
- **Status workflow** : New → In Progress → On Hold → Resolved → Closed
- **Priority levels** : Low / Normal / High / Critical — color-coded in the UI via Fiori criticality
- **Assign action** — assigns a ticket to a user and auto-transitions status to In Progress
- **Close action** — requires a mandatory closing comment (enforced by RAP validation)
- **Reopen action** — reopens a closed ticket back to New status
- **Auto-timestamps** — `CreatedAt` and `UpdatedAt` managed automatically by RAP determinations
- **List Report filters** — filter by Status, Priority, Category, Assigned To
- **OData V4 API** — fully consumable REST API generated by the RAP framework

---

## Business Rules

| Rule | Implementation |
|---|---|
| Closing comment is mandatory when Status = Closed | `checkClosingComment` validation (RAP, on save) |
| `UpdatedAt` refreshes on every relevant field change | `setUpdateTimestamp` determination (RAP, on modify) |
| `CreatedAt` + `UpdatedAt` set at creation | `setTimestamps` determination (RAP, on create) |
| Assign auto-sets status to In Progress if New | `assignTicket` action logic |
| Reopen resets status to New and clears closing comment | `reopenTicket` action logic |

## Data Model

| Field | Type | Description |
|---|---|---|
| `TICKET_ID` | `SYSUUID_C22` | Primary key (UUID, auto-generated) |
| `TITLE` | `CHAR80` | Ticket title (mandatory) |
| `DESCRIPTION` | `SSTRING` | Detailed description |
| `PRIORITY` | `CHAR1` | L / N / H / C (Low / Normal / High / Critical) |
| `STATUS` | `CHAR2` | NW / IP / OH / RS / CL |
| `CATEGORY` | `CHAR20` | Free-text category |
| `ASSIGNED_TO` | `UNAME` | SAP user login |
| `CREATED_AT` | `TIMESTAMPL` | Creation timestamp (auto) |
| `UPDATED_AT` | `TIMESTAMPL` | Last update timestamp (auto) |
| `CLOSING_COMMENT` | `CHAR255` | Required when closing a ticket |
| `CREATED_BY` | `UNAME` | Creator login (auto) |

## Activation Order (ADT)

If deploying manually via ABAP Development Tools (Eclipse):

1. `ZDTF_STATUS` — Domain
2. `ZDTF_PRIORITY` — Domain
3. `ZDTF_STATUS_DE` — Data Element
4. `ZDTF_PRIORITY_DE` — Data Element
5. `ZTICKETSAP` — Table
6. `ZTICKET_I` — CDS Interface View
7. `ZTICKET_I` — Behavior Definition (generates class skeleton)
8. `ZBP_TICKET_I` — Handler Class (paste implementation)
9. `ZTICKET_C` — CDS Projection View
10. `ZTICKET_C` — Behavior Definition Projection
11. `ZTICKET_SRV_DEF` — Service Definition
12. `ZTICKET_SRV_BIND` — Service Binding → Publish → Preview
